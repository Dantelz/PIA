PREGUNTA 1:
Al insertar un Pokémon capturado, ¿cuál es la complejidad temporal de agregarlo
al EQUIPO PRINCIPAL (array) versus al INICIO de la PC (Lista Enlazada)?
───────────────────────────────────────────────────────────────────────────────

▶ Agregar al final del Equipo Principal (array / lista de Python):
  Complejidad: O(1) amortizado.

  Justificación:
  En Python, list.append() agrega el elemento al final del array dinámico
  interno. Normalmente no requiere desplazar elementos: simplemente escribe en
  la próxima posición libre y actualiza el índice de tamaño, lo que es O(1).
  El caso "amortizado" aparece porque ocasionalmente Python necesita redimensionar
  el array (duplicar capacidad), lo que cuesta O(n) en ese momento puntual. Sin
  embargo, al distribuir ese costo entre todas las inserciones, el promedio sigue
  siendo O(1) amortizado. Como además el equipo tiene máximo 6 elementos, esta
  redimensión es irrelevante en la práctica.

▶ Agregar al INICIO de la PC (Lista Enlazada Simple):
  Complejidad: O(1) estricto (no amortizado).

  Justificación:
  En una Lista Enlazada Simple con puntero a la cabeza, insertar al inicio
  consiste en:
    1. Crear un nuevo Nodo.                     → O(1)
    2. Apuntar nuevo.siguiente → cabeza actual. → O(1)
    3. Actualizar cabeza → nuevo nodo.           → O(1)
  No hay desplazamiento de elementos, ni reasignación de memoria contigua.
  Es O(1) puro sin importar cuántos Pokémon haya en la PC.

  Si se insertara al FINAL (sin puntero a la cola), sería O(n) porque habría que
  recorrer toda la lista para llegar al último nodo. En nuestra implementación
  elegimos insertar al inicio para garantizar O(1).

Conclusión comparativa:
  - Array (final): O(1) amortizado – ventajoso por acceso indexado O(1).
  - Lista Enlazada (inicio): O(1) estricto – ventajoso para inserciones frecuentes
    y colecciones de tamaño desconocido sin necesidad de memoria contigua.

PREGUNTA 2:
¿Por qué la Pokédex se implementó con un Hash Map en lugar de una lista de Python?
───────────────────────────────────────────────────────────────────────────────

▶ Hash Map (dict de Python) – Complejidad de búsqueda: O(1) promedio

  Un Hash Map aplica una función hash al ID numérico para calcular directamente
  el índice (bucket) donde está almacenado el Pokémon. Esto permite acceder al
  valor sin recorrer ninguna otra estructura:

      hash(id)  →  índice en la tabla  →  Pokémon

  Dado que los IDs de la Pokédex Nacional llegan hasta 1025 (Gen IX), buscar por
  ID en un hash map toma tiempo constante O(1) promedio, independientemente de
  cuántos Pokémon haya registrados.

  ¿Por qué "promedio"? Porque en el peor caso teórico pueden existir colisiones
  (dos claves con el mismo hash), lo que degrada a O(n). Python resuelve esto
  con open addressing + sondeo cuadrático y rehashing, manteniendo el factor de
  carga bajo para que O(1) sea el caso habitual.

▶ Lista de Python – Complejidad de búsqueda: O(n)

  Si guardáramos los Pokémon en una lista simple, buscar por ID requeriría
  recorrer elemento por elemento hasta encontrar el que coincide:

      for pokemon in lista:          # hasta n comparaciones
          if pokemon.id == id_buscado:
              return pokemon

  Con 151 Pokémon (Gen I), en promedio se hacen 75 comparaciones.
  Con 1025 Pokémon (Gen IX), en promedio se hacen 512 comparaciones.
  A mayor cantidad, peor rendimiento: O(n) crece linealmente.

▶ Comparación directa:

  | Operación       | Hash Map       | Lista          |
  |-----------------|----------------|----------------|
  | Búsqueda por ID | O(1) promedio  | O(n)           |
  | Inserción       | O(1) promedio  | O(1) al final  |
  | Uso de memoria  | Algo mayor     | Menor          |

  Para una Pokédex donde la operación más frecuente es "dado un ID, dame el
  Pokémon", el Hash Map es la estructura óptima. La lista sólo sería preferible
  si necesitáramos orden natural o no tuviéramos un identificador único como clave.

PREGUNTA 3:
Con 6.000 Pokémon en la PC, comparar Bubble Sort vs Quick Sort.
Mencionar el peor caso para ambos.
───────────────────────────────────────────────────────────────────────────────

▶ Bubble Sort

  Complejidad promedio: O(n²)
  Complejidad peor caso: O(n²)  ← cuando la lista está en orden inverso.

  Funcionamiento:
  Realiza n-1 pasadas. En cada pasada, compara pares adyacentes y los intercambia
  si están desordenados. El elemento más "pesado" "burbujea" hasta su posición
  final en cada pasada.

  Para n = 6.000:
    Comparaciones en peor caso ≈ n(n-1)/2 = 6000 × 5999 / 2 ≈ 17.997.000
    Eso son ~18 millones de comparaciones.

  Peor caso: La lista está ordenada en sentido contrario al deseado.
  Ejemplo: Si queremos A→Z y la lista está Z→A, cada elemento
  debe burbujear desde el extremo opuesto hasta su posición.

  Con la optimización de "flag de intercambio" (implementada en el código),
  el mejor caso es O(n) cuando la lista ya está ordenada, pero el peor y el
  promedio siguen siendo O(n²).

▶ Quick Sort

  Complejidad promedio: O(n log n)
  Complejidad peor caso: O(n²)  ← cuando el pivote es siempre el mínimo o máximo.

  Funcionamiento:
  Elige un pivote, particiona la lista en dos mitades (menores y mayores al
  pivote) y aplica el mismo proceso recursivamente en cada mitad. Cuando el
  pivote divide equilibradamente, la profundidad de recursión es log₂(n).

  Para n = 6.000:
    Comparaciones promedio ≈ n × log₂(n) = 6000 × 12,9 ≈ 77.400
    Eso son ~77 mil comparaciones.

  Peor caso: La lista ya está ordenada (o en orden inverso) y siempre se elige
  como pivote el último elemento. En ese caso el pivote no divide la lista en
  mitades sino en 1 y n-1, lo que produce n niveles de recursión → O(n²).
  Con n = 6.000, el peor caso de Quick Sort ≈ 18.000.000 comparaciones (igual
  que Bubble Sort). Esto se mitiga con estrategias como "pivot aleatorio" o
  "mediana de tres".

▶ Comparación final para n = 6.000:

  | Algoritmo   | Mejor caso | Caso promedio | Peor caso |
  |-------------|------------|---------------|-----------|
  | Bubble Sort | O(n)*      | O(n²)         | O(n²)     |
  | Quick Sort  | O(n log n) | O(n log n)    | O(n²)     |
  (* solo con el flag de optimización)

  Operaciones aproximadas para n = 6.000:
  - Bubble Sort promedio: ~18.000.000
  - Quick Sort promedio:  ~77.400

  Quick Sort es ~230 veces más eficiente en el caso promedio para 6.000 Pokémon.
  En la práctica, con datos reales y desordenados, Quick Sort supera ampliamente
  a Bubble Sort. Solo en listas casi ordenadas o de tamaño muy pequeño Bubble
  Sort puede competir (o ganar) por su implementación más simple y menor overhead.
